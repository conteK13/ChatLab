# RAG 이해

RAG 이해 단계에서는 LLM이 학습된 지식만으로 답하는 구조에서 벗어나, 애플리케이션이 찾아온 외부 정보를 답변 재료로 사용하는 흐름을 이해합니다. 이번 단계에서는 아직 Vector RAG를 직접 구현하지 않고, RAG가 왜 필요한지와 Tool, Agent와 어떻게 다른지 먼저 정리합니다.

### 목표

- RAG가 무엇인지 이해하기
- LLM의 학습 지식과 애플리케이션이 제공하는 외부 지식을 구분하기
- Tool과 RAG의 차이 이해하기
- RAG의 기본 흐름인 수집, 분할, 검색, 답변 생성을 이해하기
- Vector RAG, GraphRAG, Text2SQL이 어떤 문제를 해결하는지 예시로 비교하기
- 다음 단계에서 구현할 Vector RAG의 역할 정리하기

- **완료 기준** : RAG가 모델을 다시 학습시키는 것이 아니라, 질문과 관련된 외부 정보를 검색해 프롬프트에 함께 넣는 방식이라는 점을 설명할 수 있어야 합니다. 또한 Tool은 기능 실행에 가깝고, RAG는 답변에 사용할 지식 검색에 가깝다는 차이를 예시와 함께 구분할 수 있어야 합니다.

---
### RAG란 무엇인가

RAG는 Retrieval-Augmented Generation의 줄임말입니다. 한국어로 풀어보면 "검색으로 보강한 생성"에 가깝습니다.

LLM은 입력받은 메시지와 학습된 지식을 바탕으로 답변을 생성합니다. 하지만 프로젝트 문서, 회사 정책, 최신 공지, 고객별 데이터처럼 모델이 학습하지 않았거나 실행 시점에 바뀔 수 있는 정보는 기본 지식만으로 정확히 답하기 어렵습니다.

RAG는 이런 상황에서 사용자의 질문과 관련된 정보를 먼저 찾아오고, 그 정보를 LLM에게 함께 전달해 답변을 만들게 하는 방식입니다.

흐름으로 보면 다음과 같습니다.

```text
사용자 질문
  -> 관련 문서 검색
  -> 검색된 문서를 질문과 함께 LLM에 전달
  -> LLM이 검색 결과를 바탕으로 답변 생성
```

중요한 점은 RAG가 모델 자체를 다시 학습시키는 방식이 아니라는 점입니다. 외부 문서를 찾아서 프롬프트에 넣어주는 방식에 가깝습니다.

---
### 왜 RAG가 필요한가

사용자가 다음처럼 질문한다고 가정합니다.

```text
우리 프로젝트에서 Agent 단계 다음에는 무엇을 배워?
```

LLM이 이 프로젝트의 문서를 모른다면 정확히 답할 수 없습니다. 일반적인 챗봇 개발 지식에 기대어 "아마 메모리나 RAG를 배울 수 있습니다"처럼 그럴듯하게 말할 수는 있지만, 실제 `docs/lessons/index.md`에 적힌 학습 순서를 확인한 것은 아닙니다.

RAG를 사용하면 애플리케이션이 먼저 학습 문서를 검색합니다. 검색 결과에서 "5. RAG 이해", "6. Vector RAG 구현" 같은 내용을 찾은 뒤, 그 내용을 LLM에게 전달합니다. 그러면 LLM은 추측이 아니라 검색된 문서를 바탕으로 답변할 수 있습니다.

즉 RAG의 핵심은 모델이 모르는 지식을 외워 넣는 것이 아니라, 답변 시점에 필요한 지식을 찾아서 보여주는 것입니다.

---
### Tool과 RAG의 차이

Tool과 RAG는 둘 다 LLM이 혼자 처리하기 어려운 일을 애플리케이션 도움으로 해결한다는 점에서 비슷합니다. 하지만 목적이 다릅니다.

Tool은 기능을 실행하는 방법입니다.

- 현재 시간 조회
- 날씨 API 호출
- 이메일 발송
- 계산 실행
- 데이터 저장
- 외부 시스템에 작업 요청

RAG는 답변에 필요한 지식을 찾아오는 방법입니다.

- 프로젝트 문서 검색
- 사내 정책 검색
- 제품 매뉴얼 검색
- 고객 문의 이력 검색
- 논문이나 보고서 내용 검색

예를 들어 "현재 서울 날씨 알려줘"는 날씨 API를 실행해야 하므로 Tool에 가깝습니다. 반면 "이 프로젝트에서 Tool 만들기 단계의 완료 기준이 뭐야?"는 문서에서 관련 내용을 찾아 답해야 하므로 RAG에 가깝습니다.

둘을 함께 사용할 수도 있습니다. Agent가 사용자의 요청을 보고, 날씨가 필요하면 날씨 Tool을 호출하고, 프로젝트 문서가 필요하면 RAG 검색 Tool을 호출하는 식입니다. 이때 Agent는 실행 흐름을 판단하고, Tool과 RAG는 각각 기능 실행과 지식 검색을 담당합니다.

---
### RAG의 기본 흐름

RAG는 보통 두 단계로 나누어 생각합니다.

첫 번째는 미리 준비하는 단계입니다.

```text
문서 수집
  -> 문서를 작은 조각으로 분할
  -> 검색하기 좋은 형태로 저장
```

두 번째는 사용자가 질문했을 때 실행되는 단계입니다.

```text
사용자 질문
  -> 질문과 관련된 문서 조각 검색
  -> 검색 결과를 LLM 프롬프트에 포함
  -> LLM 답변 생성
```

문서를 작은 조각으로 나누는 이유는 전체 문서를 한 번에 넣기 어렵고, 질문과 관련 없는 내용이 너무 많이 섞이면 답변 품질이 떨어질 수 있기 때문입니다. 그래서 RAG에서는 "어떤 문서를 어떻게 쪼갤 것인가"와 "질문에 맞는 조각을 어떻게 찾을 것인가"가 중요합니다.

---
### Vector RAG 예시

Vector RAG는 문서와 질문을 벡터로 바꿔서 의미가 비슷한 문서 조각을 찾는 방식입니다.

예를 들어 문서에 다음 내용이 있다고 가정합니다.

```text
Tool은 애플리케이션이 제공하는 기능입니다.
Agent는 Tool을 언제, 어떤 순서로 사용할지 판단합니다.
```

사용자가 "Agent랑 Tool은 뭐가 달라?"라고 질문하면, 문장에 완전히 같은 단어가 없더라도 의미가 가까운 문서 조각을 찾을 수 있습니다. 이때 문서 조각과 질문을 임베딩 모델로 벡터화하고, 벡터 간 거리가 가까운 조각을 검색합니다.

다음 단계인 Vector RAG 구현에서는 이 흐름을 직접 다룹니다. 학습 문서를 읽고, 문서를 조각으로 나누고, 임베딩한 뒤, 질문과 가까운 문서 조각을 찾아 LLM 답변에 활용하는 구조로 확장합니다.

---
### GraphRAG 예시

GraphRAG는 문서에서 개체와 관계를 뽑아 그래프 형태로 연결하고, 그 관계를 활용해 답변하는 방식입니다.

Vector RAG는 의미가 비슷한 문서 조각을 찾는 데 강합니다. 하지만 질문이 여러 개체 사이의 관계를 묻거나, 여러 문서에 흩어진 연결을 따라가야 한다면 그래프 구조가 도움이 될 수 있습니다.

예를 들어 프로젝트 문서에서 다음 관계를 만들 수 있습니다.

```text
Tool 만들기 -> 다음 단계 -> Agent
Agent -> 다음 단계 -> RAG 이해
RAG 이해 -> 다음 단계 -> Vector RAG 구현
```

사용자가 "Tool 다음에 Agent를 배우고 나서 왜 RAG로 넘어가?"라고 묻는다면, GraphRAG는 학습 단계 사이의 관계를 따라가며 "Tool로 기능을 연결하고, Agent로 실행 흐름을 다룬 뒤, RAG로 외부 지식 검색을 다룬다"는 식으로 설명할 수 있습니다.

GraphRAG는 조직도, 지식 그래프, 복잡한 문서 관계처럼 연결 구조가 중요한 문제에서 특히 유용합니다.

---
### Text2SQL 예시

Text2SQL은 사용자의 자연어 질문을 SQL로 바꿔 데이터베이스를 조회하는 방식입니다.

예를 들어 사용자가 다음처럼 질문할 수 있습니다.

```text
지난달 가장 많이 질문된 주제는 뭐야?
```

이 질문은 문서 검색보다 데이터베이스 집계에 가깝습니다. 애플리케이션은 자연어 질문을 SQL로 바꾸고, 데이터베이스에서 결과를 조회한 뒤, 그 결과를 LLM에게 전달해 설명하게 만들 수 있습니다.

```text
사용자 질문
  -> SQL 생성
  -> 데이터베이스 조회
  -> 조회 결과를 바탕으로 답변 생성
```

Text2SQL은 넓게 보면 외부 데이터를 찾아 답변에 활용한다는 점에서 RAG와 비슷한 문제를 해결합니다. 다만 Vector RAG가 비정형 문서 검색에 강하다면, Text2SQL은 테이블처럼 구조화된 데이터 조회에 적합합니다.

---
### 언제 무엇을 선택할까

문제에 따라 적합한 방식이 달라집니다.

- 현재 시간, 날씨, 이메일 전송처럼 실행이 필요하면 Tool을 사용합니다.
- 문서, 매뉴얼, 정책처럼 텍스트 지식에서 관련 내용을 찾아야 하면 Vector RAG를 사용합니다.
- 사람, 제품, 문서, 개념 사이의 관계를 따라가야 하면 GraphRAG를 고려합니다.
- 주문, 매출, 로그처럼 테이블에 저장된 데이터를 조회해야 하면 Text2SQL을 고려합니다.
- 여러 방식을 상황에 맞게 고르고 순서대로 실행해야 하면 Agent가 실행 흐름을 맡을 수 있습니다.

실제 서비스에서는 한 가지 방식만 쓰기보다 여러 방식을 함께 사용하는 경우가 많습니다. 예를 들어 Agent가 질문을 보고, 문서 검색이 필요하면 Vector RAG를 사용하고, 통계 조회가 필요하면 Text2SQL을 사용하고, 외부 API 실행이 필요하면 Tool을 사용할 수 있습니다.

---
### 직접 확인하기

이번 단계에서는 새 기능을 구현하지 않고, 기존 학습 문서를 대상으로 RAG가 필요한 상황을 생각해 봅니다.

다음 질문을 예시로 볼 수 있습니다.

```text
Tool 이해 단계의 완료 기준을 알려줘.
```

현재 챗봇이 학습 문서를 검색하지 못한다면, 모델은 실제 문서 내용을 확인하지 못한 채 답할 수 있습니다. 반대로 RAG가 연결되어 있다면 `docs/lessons/02_Tool이해.md`에서 관련 내용을 찾아 근거로 사용할 수 있습니다.

다음 단계에서는 이 차이를 직접 구현합니다. 학습 문서를 검색 대상으로 만들고, 사용자의 질문과 관련된 문서 조각을 찾아 답변에 넣는 Vector RAG 흐름을 추가합니다.

---
### 자주 묻는 질문
#### RAG를 쓰면 모델이 문서를 외우는가

아닙니다. RAG는 모델을 다시 학습시키지 않습니다. 질문할 때마다 관련 문서를 검색하고, 검색된 내용을 LLM 입력에 함께 넣어 답변을 만들게 합니다.

---
#### RAG를 쓰면 항상 정확한 답변이 나오는가

항상 그렇지는 않습니다. 검색된 문서가 질문과 맞지 않거나, 문서 조각이 너무 작거나 크거나, LLM이 검색 결과를 잘못 해석하면 틀릴 수 있습니다.

그래서 RAG에서는 검색 품질, 문서 분할 방식, 프롬프트 구성, 출처 표시, 답변 검증이 중요합니다.

---
#### RAG와 검색은 같은가

같지는 않습니다. 검색은 관련 문서를 찾는 과정이고, RAG는 그 검색 결과를 LLM에게 전달해 답변을 생성하는 전체 흐름입니다.

즉 검색은 RAG의 일부입니다.

---
#### RAG와 Agent는 어떤 관계인가

RAG는 외부 지식을 찾아오는 방식이고, Agent는 여러 행동 중 무엇을 할지 판단하고 실행 흐름을 이어가는 구조입니다.

Agent가 RAG 검색을 하나의 Tool처럼 사용할 수 있습니다. 예를 들어 사용자가 문서 내용을 물어보면 Agent가 문서 검색을 실행하고, 검색 결과를 바탕으로 답변하도록 만들 수 있습니다.
